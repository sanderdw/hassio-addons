# Auto-Unhide Player in Music Assistant: A Development Story

## The Problem

When VoltViz connects to Music Assistant (MA) via the SendSpin protocol, a player is created in MA. However, this player is **hidden by default** in the MA UI. Users had to manually navigate to Settings -> Players -> select the VoltViz player -> unhide it. This friction needed to be eliminated.

## Development Iterations

### Iteration 1: Research Phase

**Goal:** Understand how MA hides players and what API to call.

**Approach:** Explored the [music-assistant/server](https://github.com/music-assistant/server) repository to find:
- `_attr_hidden_by_default = True` on SendSpin web players
- `CONF_HIDE_IN_UI` config entry stored per player
- `config/players/save` WebSocket command to change it
- Player IDs pass through unchanged from client to MA

**Key finding:** The player ID in MA is exactly the string VoltViz generates (e.g., `VoltViz-ab12cd34`).

### Iteration 2: Direct Nginx Proxy (Failed)

**Idea:** Add an nginx proxy location (`/ma-api-proxy/`) pointing to MA's API port (8095), then have the frontend connect and send the unhide command.

**Implementation:**
- `run.sh`: Derived MA API URL by replacing port 8927 with 8095
- `App.tsx`: Added `unhidePlayerInMA()` function

**Result:** `crypto.randomUUID is not a function` - the page runs over HTTP (insecure context).

### Iteration 3: Fix crypto.randomUUID (Partial Fix)

**Problem:** `crypto.randomUUID()` is only available in secure contexts (HTTPS).

**Fix:** Added fallback using `Math.random().toString(36).slice(2)`.

**Result:** Function ran without error, but MA responded with `"Authentication required. Please send auth command first."` - Schema version 29 requires auth.

### Iteration 4: Ingress Port with Auth Headers (Failed)

**Idea:** Connect to MA's ingress port (8094) and inject `X-Remote-User-ID` header via nginx for auto-authentication.

**Implementation:**
- Added `hassio_api: true` to `config.json`
- Used Supervisor API to discover MA's internal IP
- Proxied to port 8094 with auth headers

**Result:** Connection silently closed. Port 8094 is IP-restricted to the Supervisor (`172.30.32.2`) only.

### Iteration 5: HA Access Token (Failed)

**Idea:** Use the HA access token (from browser localStorage) to authenticate with MA.

**Result:** `"Invalid or expired token"` - MA uses its own JWT tokens, not HA tokens.

### Iteration 6: MA's HA Ingress Entry (Success!)

**Key insight:** The browser is already authenticated with HA. If we connect to MA through **HA's ingress proxy** (the same mechanism the MA frontend uses), authentication is handled automatically by the Supervisor injecting `X-Remote-User-ID`.

**Implementation:**
1. `run.sh` queries `http://supervisor/addons/<slug>/info` for `ingress_entry`
2. Writes the ingress path to `/usr/share/nginx/html/ma-config.json`
3. Frontend fetches `ma-config.json`, then opens WebSocket to `ws://host:8123<ingress_entry>/ws`
4. Sends `config/players/save` command (no auth needed - Supervisor handles it)

**Bug fix:** Missing trailing slash in ingress path caused 404 on WebSocket handshake.

**Result:** Player successfully unhidden automatically on connect.

## Key Technical Decisions

| Decision | Why |
|----------|-----|
| Use HA Ingress instead of direct API | Auto-authentication without managing tokens |
| Generate `ma-config.json` at startup | Frontend can't access Supervisor API directly |
| Fire-and-forget unhide call | No user feedback needed, fails silently in non-addon context |
| Trailing slash normalization | Supervisor API returns paths without trailing slash |

## Making This Fully Agentic

The development cycle above required manual steps: building Docker images, restarting the add-on, and clicking buttons in the browser. Here's how this could become a fully autonomous CI/CD loop:

### 1. Commit and Push via Git

The agent already has access to `git` commands. After making code changes:
```bash
git add src/App.tsx
git commit -m "feat: auto-unhide player in Music Assistant"
git push origin feature/auto-unhide
```

### 2. GitHub Actions Workflow

A workflow that builds and deploys on push:
```yaml
name: Build and Deploy Add-on
on:
  push:
    branches: [feature/*]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run build
      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: ghcr.io/sanderdw/hassio-addons/ha-voltviz-amd64:dev
```

The agent could monitor this workflow:
```bash
gh run watch --exit-status
```

### 3. Trigger Add-on Update via HA Supervisor API

Once the image is pushed, trigger an update on the HA instance:
```bash
curl -X POST -H "Authorization: Bearer ${HA_TOKEN}" \
  http://homeassistant.local:8123/api/hassio/addons/local_voltviz/update
```

Or restart the add-on to pick up the new image:
```bash
curl -X POST -H "Authorization: Bearer ${HA_TOKEN}" \
  http://homeassistant.local:8123/api/hassio/addons/local_voltviz/restart
```

### 4. Browser Automation for E2E Validation

With the Chrome DevTools MCP plugin, the agent can:
- Navigate to the app page
- Click "Connect" on the SendSpin dialog
- Verify no console errors
- Run `evaluate_script` to confirm the WebSocket flow completed
- Check that the player appears in MA's player list

### 5. The Fully Autonomous Loop

```
Code Change → git push → GitHub Actions builds image
     ↑                              ↓
     │                     Push to container registry
     │                              ↓
     │                     Supervisor pulls new image
     │                              ↓
     │                     Add-on restarts
     │                              ↓
     │                     Chrome DevTools MCP tests
     │                              ↓
     └──── Fix if errors ←── Report results
```

### What's Missing Today

1. **HA API access from the agent** - Need a long-lived token configured for restarting add-ons remotely
2. **Webhook to trigger rebuild** - GitHub Actions needs to notify the agent when done, or the agent polls `gh run view`
3. **Automated rollback** - If E2E tests fail, revert to the previous image tag
4. **Container registry watching** - HA Supervisor can watch for image updates with `image` field in `config.json`, but needs a semver or `dev` tag strategy

The Chrome DevTools MCP plugin is the missing piece that makes true E2E validation possible without human eyes. Combined with git automation and the Supervisor API, the entire develop-deploy-validate cycle could run in a single agent conversation.
