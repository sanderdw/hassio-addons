Auto-Unhide Player in Music Assistant

A development story: 6 iterations from concept to working solution, driven by an AI agent with browser DevTools access.

The Problem

When VoltViz connects to Music Assistant (MA) via the SendSpin protocol, a player is created but hidden by default in the MA UI. Users had to manually navigate to Settings → Players → select VoltViz → unhide it. This friction needed to be eliminated.

Architecture

Browser (HA Frontend)
  └── VoltViz iframe (via HA Ingress)
        ├── SendSpin Client ──→ nginx proxy ──→ MA Port 8927 (audio stream)
        ├── fetch ma-config.json ──→ nginx (static file from run.sh)
        └── WebSocket ──→ HA Ingress Proxy ──→ MA Port 8094 (auto-authenticated)
                              │
                    Supervisor injects
                    X-Remote-User-ID header
                              │
                    MA accepts command:
                    config/players/save
                    { hide_in_ui: false }

Iteration 5: MA's HA Ingress Entry (Almost!)

run.shApp.tsx Key insight: route through MA's own HA ingress endpoint. run.sh queries Supervisor for ingress_entry, writes to ma-config.json. Frontend connects via HA's ingress proxy.

run.shApp.tsx Normalized ingress path to always include trailing slash. WebSocket connects, MA auto-authenticates via Supervisor headers, command succeeds.

Making This Fully Agentic

The development loop above required manual steps: building Docker images, restarting the add-on, and triggering the browser test. Here's how to close the loop entirely:

The Autonomous Pipeline

Code change → git push → GitHub Actions builds image → Push to GHCR → Supervisor pulls update → Add-on restarts → Chrome DevTools MCP validates → Report / fix

1. Git Automation

git add src/App.tsx
git commit -m "feat: auto-unhide player in Music Assistant"
git push origin feature/auto-unhide

The agent has full git access. Commits, branches, and PRs can be created programmatically.

2. GitHub Actions Workflow

name: Build Add-on
on:
  push:
    branches: [feature/*]
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm run build
      - uses: docker/build-push-action@v5
        with:
          push: true
          tags: ghcr.io/sanderdw/hassio-addons/ha-voltviz-amd64:dev

The agent monitors with gh run watch --exit-status and proceeds only after a green build.

3. Trigger Add-on Restart

curl -X POST \
  -H "Authorization: Bearer ${HA_LONG_LIVED_TOKEN}" \
  http://homeassistant.local:8123/api/hassio/addons/local_voltviz/restart

Requires a long-lived access token configured for the agent. The Supervisor API handles pulling the new image and restarting.

4. E2E Validation via Chrome DevTools MCP

// Agent script (pseudo-code)
navigate_page("http://ha:8123/app/local_voltviz")
click("Connect button")
wait_for("playback controls visible")
evaluate_script("check ma-config.json exists")
evaluate_script("verify WebSocket to MA ingress works")
evaluate_script("confirm player hide_in_ui === false")

This is exactly what we did manually in this session. The Chrome DevTools MCP plugin makes it fully automatable.

What's Still Missing

  • HA API token for the agent — needed to trigger add-on restarts remotely
  • Webhook / polling bridge — GitHub Actions needs to signal completion to the agent
  • Automated rollback — if E2E tests fail, revert to previous image tag
  • Image tag strategy — use :dev for feature branches, :latest for main

Key Takeaways

LessonDetail
HA add-on auth is complex6 iterations to find the right path. Each add-on's API has different auth requirements.
Ingress is the golden pathHA's ingress proxy handles auth transparently. Use it instead of reimplementing token flows.
Browser DevTools closes the feedback loopWithout evaluate_script, we'd be guessing. With it, we could test auth flows in seconds.
Fail fast, iterate smallEach iteration tested one hypothesis. Console errors gave immediate direction for the next attempt.
Secure contexts mattercrypto.randomUUID only works over HTTPS. Always have fallbacks for HTTP environments.
